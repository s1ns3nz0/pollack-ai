# UAV ATT&CK 커버리지 갭 — 31개 미탐지 기법 대응 전략

출처: 팀 「UAV ATT&CK 매핑표 (Enterprise + ICS)」 — 15 전술 × 111 기법을 로그 테이블·
컬럼·탐지상태(✅/🔜/❌)로 매핑. 본 문서는 그중 **❌(탐지 불가) ~31개**를 성격별로
분류하고 커버 전략·우선순위를 제시한다.

> 핵심 주장: 31개는 균질한 "구멍"이 아니다. **성격이 5종류**로 갈리며, 일부는 애초에
> 갭이 아니고(침해 이전), 대부분은 *센서 1개 + 컬럼 2개*로 탐지권에 들어온다. 진짜
> "탐지 어려움"은 6~8개뿐이고 그건 예방통제로 받는다.

---

## 5가지 archetype

### A. 침해 이전 · 네트워크 밖 (≈5) — 갭이 아님, scope 밖
`T1590`(네트워크 정보수집), `T1596`(공개DB 검색), `T1587`/`T1588`/`T1608`(공격도구
개발·획득·스테이징).

공격자 **자기 인프라에서, 우리 환경에 닿기 전** 발생 → 로그가 존재할 수 없다.
- **대응**: ATT&CK 커버리지 분모에서 "pre-compromise/external"로 분리(갭으로 카운트 X).
- **보완통제**: 네트워크 격리, EASM(외부 공격표면 관리). 탐지 책임 아님.

### B. 수동 수집/읽기 — 앱 로그 부재 (≈13) — 센서 1개로 대량 전환
`T1125`(영상), `T1119/T0802`(자동수집), `T0845`(임무 추출), `T0801`/`T0868`/`T0861`
(상태 수동관찰), `T1113/T0852`(화면캡처), `T1005/T0893`(로컬파일), `T1056`(키입력),
`T1074`(스테이징), `T1560`(아카이브), `T0882`(정보절취).

조용한 "읽기" 행위라 현재 미기록. 행위 자체는 측정 가능 — **센서를 안 달았을 뿐**.
- **eBPF/Falco 파일·exec 감사**(컨테이너) → 로컬파일·삭제·스테이징·아카이브·추출 ~8개
- **VNC/noVNC 세션 바이트·지속시간 카운터** → 화면/영상/키입력 캡처 ~4개
- **대응**: ❌→🔜. 이 버킷이 노력 대비 효과 최고.

### C. 암호·터널·난독 C2 (≈4) — 내용 불가, 메타데이터로 탐지
`T1573`(암호채널), `T1572`(터널링), `T1001`(난독), `T1132`(인코딩).

설계상 콘텐츠 검사 회피 → 페이로드 시그니처는 무의미.
- **대응**: 메타데이터·행위 분석 — JA3/TLS 핑거프린트, 비콘 주기성, **엔드포인트
  화이트리스트**(`C2_Whitelisted_GCS_List` 재활용), MAVLink 필드 엔트로피 이상.
- "채널의 존재·이상"을 탐지하지 내용을 탐지하지 않는다.

### D. 미계측 매체/채널 유출 (≈2) — 컬럼 2개면 해결
`T1011`(SATCOM 별도매체 — 링크에 용량 컬럼 부재), `T1567`(REST 웹서비스 — 페이로드
감사 부재).
- **대응**: 텔레메트리 2개 추가 — `UAVSatcomLink_CL.TxBytes`, C4I/stub REST 송신
  바이트 감사. 시뮬팀(uav-sim-env)에 요청할 가장 싼 항목.

### E. 파괴·루트킷·펌웨어모드 (≈6) — 예방 + 변조탐지 로깅
`T1014`/`T0851`(루트킷), `T0809`/`T1485`(데이터 파괴), `T0800`(펌웨어 업데이트 모드),
`T0878`(경보 억제).

탐지 신뢰도가 본질적으로 약함.
- **대응**: 예방통제로 전환 + 간접 탐지.
  - 시큐어부트 attestation(루트킷·펌웨어모드)
  - **호스트 밖 불변(WORM) 로그 적재** → 삭제 시 "로그 공백"으로 역탐지(파괴)
  - RTL/LAND 트리거 무력화는 결과 신호(고도·모드)로 간접 포착
- 진짜 "수용/예방으로만"인 잔여 갭.

## 우선순위 (노력 대비 효과)

| 순위 | 액션 | 전환 | 비용 |
|---|---|---|---|
| 1 | A를 "scope 밖"으로 재분류 | ~5 제거 | 0 (분류만) |
| 2 | eBPF/Falco 파일·exec 감사 센서 1개 | B ~8 | 낮음 |
| 3 | 컬럼 2개(SATCOM bytes, REST 감사) | D 2 | 매우 낮음 |
| 4 | VNC 세션 카운터 | B ~4 | 낮음 |
| 5 | C 행위분석 + E 예방통제 | 잔여 ~10 (탐지/수용 혼합) | 중간 |

**순효과**: ❌ 31 → 진짜 "수용/예방으로만"은 **6~8개**. 나머지는 센서 1 + 컬럼 2 +
행위분석으로 탐지권 진입.

## 차별점 — kill-chain 인접 추론 (coverage-by-adjacency)

개별 기법이 안 보여도 **킬체인 위 탐지 가능한 이웃들 사이에 끼어** 있다.

예: `Data Staged`(❌)·`Archive`(❌)는 보이지 않지만, 앞의 `Automated Collection`과
뒤의 `Exfiltration Over C2`(✅ `TxBytes`) 사이에 있다. **양 끝(✅)을 잡고 그래프
인접성으로 가운데(❌)를 추론**하면 보이지 않는 노드도 확률적으로 커버된다.

이 추론은 GraphRAG TTP 그래프(`data/mitre_attack_graph.yaml` + `data/attack_coverage.yaml`)
가 수행한다 — 같은 전술의 탐지 가능한 형제 기법을 *inference anchor*로 제시(`tools/
coverage.py`의 `inference_anchors`). 즉 갭 기법을 만났을 때 SOC가 "이건 직접 안 보이나,
같은 전술의 X·Y가 보이므로 그 활성 여부로 정황 추정"을 할 수 있다.

## 운영 연결

- **커버리지 KPI**: `tools/coverage.py`가 전술별 covered/planned/uncovered와 archetype별
  갭 수를 산출 → 모니터링 대시보드(로드맵 5단계)로 노출.
- **자기개선 루프**: 새 센서/룰이 들어오면 `attack_coverage.yaml` 상태를 ❌→🔜→✅로
  갱신 → 커버리지 추세가 KPI로 추적됨.
- **발표**: "31개 갭"이 아니라 "5개 archetype, 각각의 대응과 잔여 위험"으로 설명 →
  방어 성숙도를 정량·정성 동시에 입증.
